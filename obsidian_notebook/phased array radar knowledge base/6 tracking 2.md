
# 雷达追踪入门 + 代码讲解

## 一、先建立直觉：雷达追踪到底在干嘛？

雷达每隔一段时间扫描一次（这一次扫描叫一个 **dwell**），每次扫描会"看到"几个回波点（**detection**），比如"距离 1000 米、方位角 0°的地方有个东西"。

但雷达本身只看到孤立的点，它不知道:

- 这一次的点和上一次的点是同一个飞机吗？
- 这是真目标还是噪声？
- 它正在朝哪个方向飞？速度多少？

**追踪算法（tracker）的任务**就是：把多个扫描里"属于同一个目标"的点串成一条线（一条 **track / 航迹**），并估计这个目标的位置和速度。

几个会反复出现的术语：

|术语|含义|
|---|---|
|dwell|一次扫描周期|
|detection|一个扫描里看到的一个回波点|
|track|算法对一个目标的持续估计|
|state|这里指 `[x, y, vx, vy]`，平面位置 + 平面速度|
|tentative|刚刚起航的"暂定"航迹（可能是假目标）|
|confirmed|连续命中后"确认"的航迹|

## 二、代码里的数据表示

### `polar_to_xy`

雷达原始数据是极坐标（距离 + 方位角），但做运动学（位置 / 速度）在直角坐标里更舒服，所以一进来就转一下：

```
x = range * cos(az)
y = range * sin(az)
```

### `make_detection`

没什么花样，就是把一次检测打包成一个字典，同时存极坐标和转换后的 xy。

### track（航迹）的字段

代码里没有专门定义类，每条航迹是一个 dict，关键字段：

- `state`: `[x, y, vx, vy]`，4 维状态向量
- `status`: `"tentative"` 或 `"confirmed"`
- `hits` / `misses`: 累计命中数 / 累计漏失数
- `time_s`: 这条航迹"目前推到了哪个时刻"
- `last_detection_time_s`, `last_measurement_xy`: 上一次真正测到的时间和位置（用来估计速度）

## 三、追踪的核心循环：Predict → Associate → Update

这是几乎所有追踪器的共同骨架。每来一个 dwell，对每条航迹按下面三步走。

### Step 1: Predict（预测）— `predict_state`

假设目标做**匀速直线运动**（叫 constant velocity model，是最简单的运动模型）。距离上次更新过了 `dt` 秒，那就把位置往前推一推：

```python
new_x = old_x + vx * dt
new_y = old_y + vy * dt
# vx, vy 保持不变
```

预测只是"猜"，还没看到新数据。预测的目的是把航迹推到当前 dwell 的时刻，方便和这次的检测点比较。

### Step 2: Associate（关联）— `nearest_detection`

预测完了，本次有一堆检测，每条航迹要决定：**这堆点里哪个是"我"？**

代码里用的策略叫 **gating + nearest neighbor**：

1. 在预测位置周围画一个半径 `gate_m = 250` 米的圈（这个圈叫 **gate**，门）
2. 圈外的点直接不考虑（可能是别的目标或噪声）
3. 圈内最近的那个点就认为是"我"

如果一条航迹"领走"了某个检测，其他航迹就不能再用它了 —— 这就是 `used_detection_indices` 的作用，避免两条航迹抢同一个点。

> 顺带一提：如果两条航迹都在某个点附近，"先到先得"会出错。真实系统会用 GNN（global nearest neighbor，全局最优分配）或 JPDA 这种更好的算法。本代码是入门版本。

### Step 3: Update（更新）— `update_state`

找到了对应的检测，现在要把"预测值"和"测量值"融合一下。代码用的是 **α-β 滤波**（卡尔曼滤波的简化版本）。

**位置融合**：

```python
new_pos = (1 - α) * predicted_pos + α * measured_pos
```

`α = 0.7` 表示"7 成相信测量，3 成相信预测"。如果你的雷达测量噪声大，应该把 α 调小，多依赖预测。

**速度估计**：先用两次测量算"瞬时测量速度"：

```python
measured_velocity = (本次测量位置 − 上次测量位置) / 两次测量时间差
```

然后再融合：

```python
new_velocity = (1 - β) * old_velocity + β * measured_velocity
```

`β = 0.6`。代码里变量名写成 `alpha_velocity`，但概念上就是 β。

α、β 都在 (0, 1) 之间，越大越信任新数据，越小越平滑（也越滞后）。

## 四、航迹管理：起新、确认、删除

光有 predict-associate-update 还不够，还得管理航迹的"生老病死"：

**起新航迹（Step 3 in code）**：本次没被任何航迹关联上的检测，可能是新出现的目标，给它起一条新航迹，状态 `tentative`，速度初始化为 0（因为只有一个点，还估不出速度）。

**确认 / 删除**：

- 命中一次：`hits++`, `misses = 0`，如果 `hits ≥ confirmation_hits=2`，状态升级为 `confirmed`
- 漏失一次：`misses++`，如果 `misses > max_misses=1`，删除该航迹

为什么要"暂定 → 确认"两个状态？因为单个噪点也可能起一条假航迹，强制它连续命中几次再确认，可以滤掉绝大多数偶发噪声。

## 五、跟着测试数据走一遍

测试数据里有两个目标：

- **A**：起点 `(range=1000, az=0°)` → xy ≈ `(1000, 0)`，每秒 range 增加 20，往远离雷达方向飞
- **B**：起点 `(range=1400, az=20°)` → xy ≈ `(1316, 479)`，每秒 range 减 30，朝雷达方向飞

中间两个 dwell 没有 B 的检测（模拟 B 暂时丢失）。

|Dwell|t|发生了什么|
|---|---|---|
|1|0s|没有已存在的航迹。A、B 都被当作"新检测"，分别起航迹 #1、#2，都是 `tentative`，`hits=1`|
|2|1s|预测：两条航迹速度都是 0，预测位置=上次位置。A 的新检测离航迹 #1 很近（远离方向偏移约 20 米），进入 250 米的门 → 关联成功。B 同理。更新后两条航迹的 `hits=2` → 都升级为 `confirmed`。速度开始有了非零估计|
|3|2s|两条航迹继续命中，速度估计随 α-β 滤波收敛得更准|
|4|3s|只有 A 的检测。航迹 #1 命中。航迹 #2 在 gate 内找不到检测 → `misses=1`（还没超过 `max_misses=1`，暂时保留）|
|5|4s|还是只有 A。航迹 #1 继续命中。航迹 #2 又没命中 → `misses=2` > 1 → **删除航迹 #2**|

最终结果：剩下航迹 #1，对 A 的位置和速度有稳定估计。

## 六、代码里值得注意的小细节

1. 文件顶部 `import matplotlib.pyplot as plt` 但你贴的代码里没用到 —— 推测后面应该有画图代码（`track_snapshots` 和 `detection_points` 这两个列表就是为可视化准备的，但循环里只在收集，没在画）。
    
2. `dt_since_last_detection = max(1e-9, ...)`：防止两个测量时间戳相同导致除以 0。
    
3. 起新航迹时 `last_measurement_xy` 设为本次测量位置，速度设 0。下一次更新时算的"测量速度"才有意义（两点之间才能算速度）。
    
4. `predict` 里的 `dt = max(0.0, time_s - track["time_s"])`：防止时间戳乱序导致负数 dt。

## 七、如果想继续深入

这份代码是非常入门的版本。如果你想往下学，可以按这个顺序：

1. **卡尔曼滤波**（Kalman Filter）：α-β 滤波是它在常加速假设下的稳态特例。学了 KF 之后你会理解 α、β 应该怎么从噪声方差推出来，而不是手调。
2. **更好的关联算法**：GNN（匈牙利算法做全局最小代价分配）、PDA、JPDA、MHT。
3. **更复杂的运动模型**：CV（匀速）→ CA（匀加速）→ IMM（多模型自适应，能处理转弯）。
4. **杂波建模、轨迹起始策略**（M/N logic 比这里的简单 hits/misses 更鲁棒）。

任何一步想展开讲都可以告诉我。
